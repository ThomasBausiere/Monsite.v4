Introduction

1. Qu'est-ce que le contrôle de version?

Le contrôle de version est un système qui enregistre les modifications apportées à un fichier ou à un ensemble de fichiers au fil du temps. Cela permet de rappeler des versions spécifiques ultérieurement.

1.1 Pourquoi le contrôle de version est-il important?

  1. Permet de revenir à un état précédent.
  2. Collaborez avec d'autres développeurs sans écraser le travail de chacun.
  3. Suivez et comprenez l'historique du projet.

2. Historique de Git

2.1. Origines et développement

Git a été initialement conçu et développé par Linus Torvalds pour le développement du noyau Linux. Avant l'introduction de Git, le noyau Linux utilisait un système appelé BitKeeper. Cependant, en raison de restrictions sur l'utilisation gratuite de BitKeeper, la communauté Linux a cherché une alternative. En 2005, Linus a décidé de créer son propre système: Git.

Le nom "Git" est d'origine britannique, qui signifie "idiot" ou "stupide". C'était une blague auto-dépréciative de la part de Linus, suggérant que Git n'était qu'un "stupide système de suivi de contenu".

L'objectif principal de Git était de créer un outil de contrôle de version qui était:

  1. Rapide : La performance était une préoccupation majeure. Les opérations devraient être effectuées aussi rapidement que possible, même sur des dépôts de grande taille.
  2. Distribué : Contrairement à la plupart des autres systèmes de contrôle de version de l'époque qui étaient centralisés, Git est distribué. Cela signifie que chaque développeur possède une copie locale complète du dépôt avec l'historique complet, permettant des opérations sans la nécessité d'une connexion réseau.
  3. Support pour le développement non-linéaire : Git a été conçu avec un excellent support pour les branches, permettant un développement parallèle sur plusieurs fonctionnalités.

Au fil des ans, Git a évolué et s'est amélioré grâce aux contributions de nombreux développeurs. Il est aujourd'hui utilisé par des millions de projets à travers le monde.

2.2. Pourquoi Git?

La montée en popularité de Git n'était pas uniquement due à son origine avec le noyau Linux. Voici quelques raisons pour lesquelles Git est devenu le système de contrôle de version préféré pour tant de développeurs:

  1. Performance : Git a été conçu pour être rapide. Cela le rend efficace même avec de grands dépôts et des historiques complexes.
  2. Flexibilité : La nature distribuée de Git donne aux développeurs une grande flexibilité. Ils peuvent travailler localement, expérimenter sans risque et seulement partager leurs changements lorsqu'ils sont prêts.
  3. Intégrité : Git utilise un mécanisme de cryptage SHA-1 pour garantir l'intégrité des données. Cela signifie qu'il est pratiquement impossible de modifier l'historique des révisions sans que cela ne soit détecté.
  4. Branching et fusion simples : Les branches sont au cœur de Git. Cela rend le processus de création, de fusion et de suppression des branches extrêmement simple et rapide.
  5. Communauté solide et support : Étant open-source, Git bénéficie d'une grande communauté qui contribue continuellement à son développement, fournit des mises à jour, des corrections de bugs et offre une vaste gamme de ressources pour l'apprentissage.

L'arrivée de plateformes telles que GitHub, GitLab et Bitbucket a également contribué à la popularité de Git, en offrant des outils et services centralisés pour faciliter la collaboration et le partage de code entre développeurs du monde entier.

3. Principes de base de Git

La maîtrise de Git commence par la compréhension de ses principes fondamentaux. Ces concepts sont les piliers du système de contrôle de version Git, et une fois que vous les maîtrisez, vous trouverez que Git est à la fois puissant et intuitif.

3.1. Repositories

Un dépôt Git, souvent appelé "repo", est un conteneur pour un projet qui suit l'historique des changements du projet. C'est l'endroit où Git garde toutes les métadonnées et les informations nécessaires pour les versions du projet. Il existe deux types principaux de dépôts :

  1. Local Repository : Il s'agit de la version du dépôt qui réside localement sur la machine de l'utilisateur. C'est ici que tous les fichiers, commits, et branches sont stockés.
  2. Remote Repository : Il s'agit de la version du dépôt qui est hébergée sur un serveur distant, généralement sur des plateformes comme GitHub, GitLab, ou Bitbucket. C'est l'endroit où les équipes collaborent.

3.2. Commits

Un commit représente une "instantané" de votre projet à un moment donné. Chaque commit est essentiellement une capture de l'état du projet, accompagnée d'un message qui décrit ce qui a été modifié ou ajouté. Pensez-y comme une photo à un moment précis de votre projet.

Chaque commit a :

  1. Un identifiant unique (SHA-1 hash) qui le représente.
  2. Une référence aux commits parents (habituellement un, mais peut être plusieurs dans le cas d'une fusion).
  3. Un message de commit.

Le processus typique implique de faire des changements, de "stager" ces changements avec git add, puis de les "commiter" avec git commit.

3.3. Branches

Les branches sont l'une des fonctionnalités les plus puissantes de Git. Elles représentent une ligne de développement indépendante. Pensez à une branche comme une manière d'isoler votre travail. Par exemple, vous pourriez avoir une branche pour développer une nouvelle fonctionnalité, une autre pour corriger un bug, etc.

  1. Master/Main Branch : Il s'agit de la branche par défaut lorsque vous initialisez un dépôt. Elle est généralement utilisée pour représenter la version stable et en production du projet.
  2. Feature Branches : Ces branches sont généralement créées pour développer de nouvelles fonctionnalités.

Créer, basculer et fusionner des branches avec Git est un processus facile et rapide.

3.4. Merge & Rebase

Ces deux opérations sont utilisées pour intégrer les changements d'une branche à une autre.

  • Merge : Prend les commits d'une branche et les intègre sur une autre. Si des changements ont été faits sur les deux branches, Git créera un nouveau "commit de fusion" pour combiner les deux historiques.
  • Rebase : Plutôt que de combiner les commits des deux branches avec un commit de fusion, le rebase prend tous les commits d'une branche et les "rejoue" sur une autre. Cela crée un historique linéaire, mais il faut être prudent avec cette méthode, surtout lorsqu'on travaille sur des branches partagées.

4. Installation et Configuration

L'une des premières étapes pour commencer avec Git est de l'installer sur votre machine. Une fois installé, il est essentiel de le configurer correctement pour s'assurer que vos commits reflètent correctement votre identité et pour optimiser votre expérience utilisateur.

4.1. Installation de Git

Sur windows

  1. Visitez le site officiel de Git à l'adresse https://git-scm.com/ et téléchargez l'installeur pour Windows.
  2. Exécutez le fichier .exe téléchargé.
  3. Suivez les instructions à l'écran en utilisant les paramètres par défaut pour la plupart des installations.

Sur Mac

  1. Si vous avez installé Homebrew, exécutez la commande : brew install git.
  2. Sinon, allez sur https://git-scm.com/ et téléchargez le package macOS, puis suivez les instructions à l'écran.

Sur Linux: Debian/Ubuntu

  1. Ouvrez un terminal.
  2. Exécutez la commande : sudo apt-get update
  3. Ensuite : sudo apt-get install git

Sur Linux: Fedora

  1. Ouvrez un terminal.
  2. Exécutez la commande : sudo dnf install git

4.2. Configuration initiale (username, email)

Une fois Git installé, il est recommandé de définir votre nom d'utilisateur et votre adresse e-mail. Ces informations seront utilisées pour associer vos commits à votre identité.

  1. Définir votre nom : Ouvrez le terminal et tapez :
    git config --global user.name "Votre Nom"
  2. Définir votre e-mail : Dans le terminal, tapez :
    git config --global user.email "votre.email@example.com"

Pour vérifier vos configurations :

git config --list

Cette commande affichera toutes les configurations de Git, y compris le nom d'utilisateur et l'e-mail que vous venez de définir.

5. Commandes Git essentielles

Git, avec sa flexibilité et sa puissance, a une variété de commandes qui permettent aux développeurs de gérer et de naviguer dans leur historique de code. Chaque commande a son propre but et permet de manipuler le code de différentes manières.

5.1. git init, clone

  • git init : Cette commande est utilisée pour initialiser un nouveau dépôt Git. Elle crée un nouveau sous-répertoire .git contenant tous les fichiers nécessaires pour la gestion de version.
    git init
  • git clone : Cette commande est utilisée pour cloner (ou copier) un dépôt à partir d'une adresse existante.
    git clone [url-du-depot]

5.2. git add, commit

  • git add : Elle ajoute des changements dans le répertoire de travail à la zone d'attente. Elle indique à Git que vous voulez inclure les mises à jour de ces fichiers dans le prochain commit.
    git add [nom-du-fichier]
  • git commit : Elle capture l'état actuel de la zone d'attente dans un nouveau commit. Ce commit est ajouté à la branche actuelle.
    git commit -m "Message du commit"

5.3. git push, pull, fetch

  • git push : Elle envoie les commits locaux à la branche principale du dépôt distant.
    git push [nom-du-depot-distant] [nom-de-la-branche]
  • git pull : Elle récupère les modifications d'un dépôt distant et les fusionne avec la branche locale.
    git pull [nom-du-depot-distant] [nom-de-la-branche]
  • git fetch : Elle récupère les changements d'un dépôt distant, mais contrairement à pull, elle ne fusionne pas ces changements avec la branche actuelle.
    git fetch [nom-du-depot-distant]

5.4. git branch, checkout

  • git branch : Elle affiche la liste des branches dans le dépôt et montre la branche actuellement active.
    git branch
  • git checkout : Utilisée pour basculer entre différentes branches.
    git checkout [nom-de-la-branche]

5.5. git merge, rebase

  • git merge : Elle fusionne les modifications d'une autre branche avec la branche actuelle. Elle est souvent utilisée pour combiner les branches de fonctionnalités avec la branche principale.
    git merge [nom-de-la-branche]
  • git rebase : Elle déplace ou combine une séquence de commits vers une nouvelle base de commit. Elle est utilisée pour maintenir un historique propre.
    git rebase [nom-de-la-branche]

5.6. git stash, reset, log

  • git stash : Elle met temporairement de côté (ou "cache") les modifications que vous avez faites mais que vous ne voulez pas encore commit. C'est utile si vous avez besoin de changer de branche mais que vous ne voulez pas commit vos changements actuels tout de suite.
    git stash
  • git reset : Elle réinitialise votre HEAD à l'état spécifié, soit en effaçant des commits, en déplaçant la pointe d'une branche vers un commit particulier, ou en sortant des changements de la zone d'attente.
    git reset --hard [nom-du-commit]
  • git log : Elle affiche l'historique des commits pour la branche actuellement active.
    git log

6. Travailler avec GitHub

Alors que Git est un système de contrôle de version permettant de suivre et de gérer les modifications du code, GitHub est une plateforme en ligne qui utilise Git pour la collaboration et la versioning. C'est un lieu où les développeurs peuvent héberger, partager et collaborer sur des projets.

6.1. Qu'est-ce que GitHub?

GitHub est une plateforme de développement collaborative qui facilite le partage et la collaboration sur des projets utilisant Git. C'est non seulement un lieu pour stocker des répertoires publics et privés, mais aussi une plateforme puissante offrant une multitude de fonctionnalités pour les développeurs, telles que la gestion des tâches, la collaboration en équipe et l'intégration continue.

6.2. Création d'un compte et configuration

  1. Inscription :

    • Visitez https://github.com.
    • Cliquez sur "Sign up" et suivez les instructions pour créer un compte.

  2. Configuration :

    • Une fois inscrit, configurez votre profil. Ajoutez une photo de profil, une bio et d'autres informations pertinentes.
    • Configurer également vos paramètres de notification, de sécurité et de confidentialité selon vos préférences.

6.3. Pusher un repository vers GitHub

  1. Créez un nouveau dépôt sur GitHub en cliquant sur le signe "+" en haut à droite et sélectionnez "New repository".
  2. Nommez votre dépôt, ajoutez une description et choisissez entre un dépôt public ou privé.
  3. Dans votre terminal ou invite de commande, naviguez vers votre projet Git local.
  4. Liez votre dépôt local à votre dépôt GitHub en utilisant la commande :
    git remote add origin [URL-du-depot-GitHub]
  5. Poussez vos commits vers GitHub avec :
    git push -u origin master

6.4. Cloner un repository depuis GitHub

Pour copier un dépôt GitHub sur votre machine locale :

git clone [URL-du-depot-GitHub]

6.5. Les Pull Requests

Les Pull Requests (PR) sont un mécanisme par lequel un développeur peut demander que les modifications qu'il a apportées soient intégrées (ou "fusionnées") dans une branche principale ou d'autres branches.

  1. Créez une nouvelle branche et effectuez vos modifications.
  2. Poussez cette branche sur GitHub.
  3. Sur GitHub, cliquez sur "New Pull Request" et choisissez la branche que vous avez poussée.
  4. Ajoutez une description détaillée de vos modifications, puis cliquez sur "Create Pull Request".

6.6. La collaboration: Issues, Labels et Milestones

  • Issues : Ce sont des propositions, des tâches, des bugs ou des demandes de fonctionnalités qui peuvent être discutées et assignées à des membres spécifiques de l'équipe.
  • Labels : Ils peuvent être ajoutés aux issues ou aux PR pour catégoriser et filtrer rapidement par type, priorité, etc.
  • Milestones : Ce sont des collections d'issues qui correspondent à un projet, une fonctionnalité ou une période de temps. Ils peuvent aider à suivre le progrès et les échéances des tâches.

7. Bonnes pratiques

L'utilisation efficace de Git nécessite plus que la simple connaissance des commandes. Adopter de bonnes pratiques peut grandement améliorer la clarté et la maintenabilité de vos projets, tout en facilitant la collaboration avec d'autres développeurs.

7.1. Messages de commit

Des messages de commit clairs et descriptifs sont essentiels pour comprendre l'historique d'un projet. Voici quelques recommandations pour rédiger des messages efficaces :

  1. Utilisez le format présent-titresque : Commencez vos messages par un verbe à l'infinitif comme "ajouter", "corriger" ou "retirer".
  2. Limitez la longueur : Le titre de votre commit ne doit pas dépasser 50 caractères. Si une description plus détaillée est nécessaire, laissez une ligne vide après le titre et ajoutez une description plus détaillée en dessous.
  3. Soyez précis : Le message doit décrire ce que le commit fait et pourquoi, et non comment il le fait..
  4. Évitez les messages vagues comme "corrections diverses".

7.2. GitFlow

GitFlow est une stratégie de branching qui définit une structure standard pour les branches, facilitant la collaboration et le déploiement.

  1. Master : Cette branche contient le code de production. Chaque commit sur cette branche est une nouvelle version en production.
  2. Develop : C'est la branche de développement principale. Les nouvelles fonctionnalités sont fusionnées ici.
  3. Feature branches : Ces branches sont créées pour développer de nouvelles fonctionnalités. Une fois terminées, elles sont fusionnées dans la branche 'develop'.
  4. Release branches : Prêtes pour la production, elles contiennent des fonctionnalités qui sont testées et prêtes à être fusionnées dans 'master'.
  5. Hotfix branches : Utilisées pour appliquer rapidement des correctifs à la production.

7.3. Git hooks

Les hooks Git sont des scripts qui peuvent être exécutés automatiquement à chaque fois qu'un événement spécifique se produit dans un dépôt. Ils sont utiles pour automatiser des tâches telles que le linting, les tests unitaires, ou la mise en forme automatique du code.

  1. Installation : Les hooks résident dans le répertoire .git/hooks de votre dépôt. Pour les activer, vous devrez les renommer (en retirant .sample par exemple) et les rendre exécutables.
  2. Hooks courants :
    • pre-commit : S'exécute avant qu'un commit soit créé. Il est couramment utilisé pour exécuter des tests ou vérifier le formatage du code.
    • post-commit : S'exécute après qu'un commit a été créé.
    • pre-push : S'exécute avant qu'un push soit effectué vers le dépôt distant.

8. Résoudre les conflits

L'un des défis majeurs lors de l'utilisation de Git, en particulier dans un environnement collaboratif, est la gestion des conflits. Ces conflits surviennent généralement lorsque plusieurs personnes travaillent sur le même code et essaient de fusionner leurs modifications.

8.1. Qu'est-ce qu'un conflit?

Un conflit se produit lorsque deux branches modifient le même morceau de code et que Git ne sait pas comment combiner ces modifications. Git marquera le fichier comme étant en conflit et il vous incombera de le résoudre.

8.2. Identifier un conflit

Lorsque vous essayez de fusionner deux branches et qu'un conflit se produit, Git vous avertira via la ligne de commande :

CONFLICT (content): Merge conflict in [nom-du-fichier]
Automatic merge failed; fix conflicts and then commit the result.

Dans le fichier en conflit, Git ajoutera des marqueurs pour aider à identifier la zone problématique :

<<<<<<< HEAD
[Votre code original]
=======
[Code modifié provenant de l'autre branche]
>>>>>> [hash-du-commit]

8.3. Résolution manuelle des conflits

  1. Ouvrez le fichier en conflit : Utilisez votre éditeur de code pour ouvrir le fichier présentant le conflit.
  2. Localisez les marqueurs de conflit : Recherchez les sections du fichier délimitées par <<<<<<<, =======, et >>>>>>.
  3. Décidez de la modification à conserver : Supprimez les marqueurs de conflit et choisissez entre votre modification, la modification de l'autre branche, ou combinez les deux selon ce qui est logique pour votre projet.
  4. Testez vos modifications : Assurez-vous que le code fonctionne comme prévu et qu'il ne casse rien d'autre.
  5. Committez les changements : Une fois que vous avez résolu tous les conflits, ajoutez les fichiers modifiés avec git add et créez un nouveau commit.

8.4. Outils de résolution de conflits

Il existe plusieurs outils graphiques qui peuvent aider à résoudre les conflits. Certains éditeurs de code et environnements de développement intégrés (IDE) possèdent des résolveurs de conflits intégrés. Parmi les outils populaires, on trouve :

  • Visual Studio Code : Il a une excellente intégration de Git et fournit une interface visuelle pour résoudre les conflits.
  • KDiff3, Meld, Beyond Compare : Des outils spécifiquement conçus pour comparer et fusionner des fichiers.

9. Outils annexes et intégrations

La puissance de Git et GitHub ne se limite pas à la ligne de commande. Il existe une pléthore d'outils et d'intégrations qui renforcent et simplifient votre expérience avec ces systèmes.

9.1. GUI pour Git

Bien que la ligne de commande soit un moyen puissant d'interagir avec Git, beaucoup préfèrent une interface graphique pour visualiser et gérer leurs dépôts.

  1. GitHub Desktop : Une interface simple et directe développée par GitHub. Elle offre des fonctionnalités de base, comme le commit, le push, et le pull, et intègre également un historique visuel des commits.
  2. SourceTree : Développé par Atlassian, cet outil offre une interface riche pour gérer vos dépôts. Il prend en charge plusieurs dépôts et a une intégration solide avec Bitbucket en plus de GitHub.
  3. TortoiseGit : C'est une extension pour Windows qui intègre Git dans l'explorateur de fichiers, permettant des opérations directement depuis le contexte des dossiers et fichiers.

9.2. Intégration avec des IDE

Les environnements de développement intégrés (IDE) modernes viennent souvent avec une intégration Git. Cette intégration facilite le suivi des changements, la gestion des branches, et l'exécution des commandes Git sans quitter l'IDE.

  1. Visual Studio Code (VSCode) : Avec son extension Git intégrée, VSCode permet de voir les changements, de gérer les branches, et d'exécuter la plupart des commandes Git directement depuis l'éditeur.
  2. IntelliJ IDEA : Cet IDE propose une intégration Git riche, avec un historique visuel, une résolution de conflits graphique, et une prise en charge de multiples dépôts.
  3. Eclipse : Via le plugin EGit, Eclipse offre une intégration complète avec Git, incluant un historique, une gestion des branches, et bien plus.

9.3. CI/CD avec GitHub Actions

L'intégration continue (CI) et la livraison continue (CD) sont devenues des éléments centraux du développement moderne. GitHub Actions offre une manière puissante d'automatiser ces processus.

  1. Configuration : Les workflows sont définis dans des fichiers .yml situés dans le répertoire .github/workflows de votre dépôt.
  2. Déclencheurs : Les workflows peuvent être déclenchés par une variété d'événements, tels que des push, des pull requests ou des releases.
  3. Jobs et étapes : Les workflows sont composés de jobs, et chaque job a des étapes. Ces étapes peuvent inclure des commandes comme l'exécution de tests, la construction d'artefacts, ou le déploiement vers des serveurs.
  4. Matrices de build : Vous pouvez définir des matrices pour exécuter des jobs sur plusieurs versions d'un langage ou sur différents systèmes d'exploitation.
  5. Stockage d'artefacts et journaux : Après l'exécution d'un workflow, GitHub Actions permet de stocker des fichiers résultants (comme des binaires) et fournit des logs détaillés de chaque étape.

10. Conclusion

Au fil de ce cours, nous avons exploré les nombreuses facettes de Git et GitHub. De ses origines historiques à sa puissante intégration avec d'autres outils, Git est devenu l'outil incontournable de versioning pour les développeurs du monde entier.

Git n'est pas seulement un outil pour suivre les modifications de code. C'est une philosophie de collaboration, permettant aux équipes de travailler ensemble de manière transparente, de gérer les conflits et de construire des logiciels robustes.

L'importance de la pratique : Comme pour toute compétence, la maîtrise de Git nécessite de la pratique. Chaque projet, chaque conflit résolu et chaque pull request fusionnée vous rapproche d'une meilleure compréhension de cet outil puissant.

L'évolution continue : Git, tout comme le monde du développement de logiciels, est en constante évolution. Il est donc essentiel de rester à jour avec les nouvelles fonctionnalités et pratiques recommandées.
